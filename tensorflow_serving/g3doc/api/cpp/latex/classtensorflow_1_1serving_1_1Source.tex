\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename T$>$\newline
class tensorflow\+::serving\+::\+Source$<$ T $>$}

An abstraction for a module that sources servables to load, or, more precisely, handles to data that can be used to load those servables. Examples of such data handles are\+:
\begin{DoxyItemize}
\item a file-\/system path to a serialized vocabulary map
\item a handle to an incoming RPC that specifies a machine-\/learned model to load
\item a \mbox{\hyperlink{classtensorflow_1_1serving_1_1Loader}{Loader}} (see \mbox{\hyperlink{loader_8h_source}{loader.\+h}}) The data handles are generally assumed to be small.
\end{DoxyItemize}

A \mbox{\hyperlink{classtensorflow_1_1serving_1_1Source}{Source}} monitors some external resource (e.\+g. file system, RPC calls) to find out about new servables and/or new versions of servables and/or the need to unload servable versions. It uses the provided callback to instruct a \mbox{\hyperlink{classtensorflow_1_1serving_1_1Target}{Target}} module (e.\+g. \mbox{\hyperlink{classtensorflow_1_1serving_1_1AspiredVersionsManager}{Aspired\+Versions\+Manager}}) which version(s) of a given servable to load. Furthermore, depending on the semantics of the \mbox{\hyperlink{classtensorflow_1_1serving_1_1Target}{Target}} module, the \mbox{\hyperlink{classtensorflow_1_1serving_1_1Source}{Source}} implicitly instructs it which ones to unload by omitting those servables.

A common case is that a \mbox{\hyperlink{classtensorflow_1_1serving_1_1Source}{Source}} emits versions for exactly one servable. An even simpler case is that a servable has a single, static version for the lifetime of the server.

Sources can house state that is shared among multiple emitted servables, e.\+g.
\begin{DoxyEnumerate}
\item A shared thread pool or other resource that multiple servables use.
\item A shared read-\/only data structure that multiple servables use, to avoid the time and space overhead of replicating the data structure in each servable instance. Shared state whose initialization time and size is negligible (e.\+g. thread pools) can be created eagerly by the source, which then embeds a pointer to it in each emitted \mbox{\hyperlink{classtensorflow_1_1serving_1_1ServableData}{Servable\+Data}} item. Creation of expensive or large shared state should be deferred to the first applicable \mbox{\hyperlink{classtensorflow_1_1serving_1_1Loader_a7dadc89ccbf488aae0102368261cc692}{Loader\+::\+Load()}} call, i.\+e. governed by the manager. Symmetrically, the \mbox{\hyperlink{classtensorflow_1_1serving_1_1Loader_addca8f4264380e5e635bbe1197f5347f}{Loader\+::\+Unload()}} call to the final servable using the expensive/large shared state should tear it down. 
\end{DoxyEnumerate}

A callback for a \mbox{\hyperlink{classtensorflow_1_1serving_1_1Source}{Source}} to supply version(s) of a servable to a \mbox{\hyperlink{classtensorflow_1_1serving_1_1Target}{Target}}, to be loaded.

A single invocation of the callback pertains to a single servable stream (given by \textquotesingle{}servable\+\_\+name\textquotesingle{}). All versions supplied in a call must be for the servable identified in \textquotesingle{}servable\+\_\+name\textquotesingle{}. Invocations on different servable streams are orthogonal to one another.

Multiple invocations may supply servable-\/data objects with identical ids (i.\+e. same servable name and version). Such servable-\/data objects are treated as semantically equivalent. The recipient will ultimately retain one and discard the rest.

If a servable version V is supplied in a first invocation, and subsequently omitted from a second invocation, the implication of omitting V depends on the semantics of the \mbox{\hyperlink{classtensorflow_1_1serving_1_1Target}{Target}} of the callback. Certain Targets will interpret V\textquotesingle{}s omission as an implicit instruction to unload V. Each \mbox{\hyperlink{classtensorflow_1_1serving_1_1Target}{Target}} must document its semantics in this regard.
